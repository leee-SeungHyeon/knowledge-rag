## Stage A — 계획

1. 요구사항을 분석하고 TodoWrite로 **3~7개** 작업 항목으로 분해한다.
2. 각 항목은 **검증 가능한 수용 기준**을 한 줄씩 포함해야 한다.
   - 예: "load_markdown_files가 빈 디렉토리에서 빈 리스트 반환 — pytest로 확인"
3. **사전 점검 — 분해 전에 다음을 명시적으로 확인한다:**
   - **런타임 의존**: 권한·쓰기 경로·환경 변수·외부 서비스 등 "구현 외 가정"이
     필요한 항목이 있는지. 있으면 별도 작업 항목으로 추가하거나 수용 기준에 반영.
     (예: non-root 컨테이너에서 쓰기 디렉토리, 포트 충돌, 시크릿 주입)
   - **의존성 그래프 / 외부 환경 한계**: 정량 기준(이미지 크기, 응답 시간 등)을
     수용 기준에 넣을 때는 라이브러리/프레임워크의 실제 규모를 한 번 확인한 뒤
     설정. 임의의 라운드 넘버(예: "300MB") 금지.
4. 분해가 끝나면 사용자 승인을 기다리지 않고 즉시 Stage B로 진행한다.

작업이 너무 크거나(7개 초과 분해 필요) 보안·아키텍처 설계가 핵심인 경우,
Stage A에서 멈추고 사용자에게 **"이 작업은 /xflow 사용을 권장합니다"**
라고 안내한 뒤 종료한다.

## Stage B — 구현

1. 계획대로 코드를 작성한다. 글로벌 `~/.claude/CLAUDE.md`의 Coding Style 준수.
2. 한 항목을 완료할 때마다 즉시 TaskUpdate(status="completed").
3. 글로벌 룰의 위험 행위 게이트(파일 삭제, force push, 의존성 제거 등)는
   여전히 적용된다 — 자동 루프가 무시하지 않는다.

## Stage C — 자가 점검

다음 3가지를 순서대로 수행한다:

1. **린트**: `ruff check` 실행.
2. **테스트**: 변경된 파일과 관련된 pytest 실행
   (예: `app/loader.py` 변경 시 `pytest tests/test_loader.py`).
3. **자문**: "Stage A에서 정의한 수용 기준을 모두 충족했는가?" 를
   각 기준에 대해 yes/no로 명시적으로 확인한다.

**실패 처리:**
- 실패 발견 시 **1회만** 수정 시도.
- 1회 수정 후에도 실패가 남으면 그대로 Stage D로 넘어가 미해결 항목으로 보고.
- 무한 루프 금지.

## Stage D — 보고

다음 형식으로 **2~4줄** 보고한다:

```
변경: <한 줄 요약>
점검: ruff <pass/fail>, pytest <N passed / M failed>, 수용 기준 <N/N>
미해결: <있으면 한 줄, 없으면 "없음">
다음 권장: <한 줄, 후속 작업이나 v2로의 승격 신호 등>
```

code-reviewer 서브에이전트는 호출하지 않는다 (v2의 영역).

## v2 승격 신호

다음 중 하나가 발생하면 사용자에게 보고한 뒤 `OBSERVATIONS.md`에 기록한다:

- v1 자가 점검을 통과했지만 사용자가 직접 버그를 발견 → 1건 기록
- 보안 또는 아키텍처 결함이 v1에서 누락됨 → **즉시 v2 승격 권고**
- 동일 종류의 누락이 3건 누적 → v2 승격 권고
