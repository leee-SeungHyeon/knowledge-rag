# RAG 아키텍처 정리

## 핵심 흐름

RAG(Retrieval-Augmented Generation)는 외부 지식을 검색해서 LLM 프롬프트에 넣어주는 패턴이다.

```
문서 → 청킹 → 임베딩 → 벡터 DB
                          ↓
질문 → 임베딩 → 유사도 검색 → 관련 청크 + 질문 → LLM → 답변
```

## 청킹 전략

- 고정 크기 청킹: 단순하지만 문맥이 잘릴 수 있음
- 재귀적 청킹: 문단 → 문장 순으로 분할하며 자연스러운 경계 유지
- 시맨틱 청킹: 임베딩 기반으로 의미가 비슷한 부분끼리 묶음

청크 크기는 보통 500~1000 토큰, 오버랩은 10~20% 정도가 일반적이다.

## 임베딩

문서와 질문을 같은 벡터 공간에 매핑한다. OpenAI의 text-embedding-3-small은
1536차원이며 가성비가 좋다. 한국어 문서가 많다면 다국어 모델을 검토해야 한다.

## 벡터 DB

- ChromaDB: 가벼움, 로컬/임베디드용
- Pinecone: 매니지드 서비스, 대규모 운영용
- Qdrant: 자체 호스팅 가능, 성능 우수

## 검색 품질 향상

- 하이브리드 검색: 벡터 유사도 + 키워드(BM25)
- 재순위(re-ranking): 1차 검색 후 cross-encoder로 정밀 재정렬
- 메타데이터 필터링: 날짜, 출처 등으로 후보 축소
